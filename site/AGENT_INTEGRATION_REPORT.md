# 百泉古建筑群导游 Agent 接入报告

调研日期：2026-03-19

## 1. 当前结论

当前网站已经适合接入导游 Agent，但最适合的落地方式不是“直接塞一个聊天框”，而是：

- 前端维持古风、自有导游 UI
- 后端接一个真实导游 API
- 先做好文本导游
- 再扩到可播放讲解
- 最后再评估实时语音

这一轮已经把前端导游骨架接上，并进一步调整成更像“随身伴游”的形态。

## 2. 已完成的接入能力

### 前端体验层

已加入这些能力：

- 灯笼式导游入口
- 切换场景时的短暂“耳语”提示
- 当前场景引导卡
- 路线化游法入口
- 行游札记区
- 自由提问折叠区

这意味着导游现在更像“陪你走园子的人”，而不是“等你敲字的客服框”。

### 数据与上下文层

前端请求时会携带：

- 当前页面类型
- 当前建筑 `spotId`
- 已访问建筑列表
- 当前建筑完整结构化数据
- 关联建筑
- 当前激活路线 `activeRouteId`
- 导游人设与规则文件内容
- 路线目录 `routeCatalog`

这已经满足“后面给一个 API 就能接起来”的目标。

## 3. 关于人设和上下文是否写入文件

结论：是，应该写到文件里。

原因很明确：

1. 方便版本化  
导游口吻、规则、路线策略会不断调，不应该散落在组件里。

2. 方便前后端共用  
前端现在已经能直接导入这些文件，后端后续也可以复制或直接读取同一份语料结构。

3. 方便做更稳定的历史语境  
后续每个建筑如果再补“短讲解词”“图像说明词”“路线导语”，也适合继续文件化。

当前已拆出的文件：

- `src/guide/prompts/persona.md`
- `src/guide/prompts/experience-rules.md`

推荐后续继续补：

- 每个建筑一份导览短讲
- 每条路线一份导语 / 收束语
- 每种图片类型一份说明模板

## 4. 这次导游为什么比普通聊天框更合适

这次不是直接做 AI 聊天，而是按“游览感”设计：

### 4.1 导游先引路，再回答

导游默认先给路线、到达提示、场景引子，而不是空白输入框。

### 4.2 用户在逛，而不是在提问

切换场景时会出现短暂耳语，像导游在前面轻轻接一句，减少用户必须主动发问的压力。

### 4.3 路线比问答更重要

对古建筑群来说，真正有意思的不只是单点知识，而是：

- 门庭如何引入湖心
- 湖心如何过渡到书院与行宫
- 山径、祠庙与碑刻如何构成另一条文脉

所以这次把路线预设直接放进导游体系里。

## 5. 研究与判断

### OpenAI 官方能力判断

OpenAI 当前更推荐用 `Responses API` 来做这类多轮、带上下文、可工具化的 Agent 交互。  
来源：

- https://developers.openai.com/api/docs/guides/migrate-to-responses
- https://developers.openai.com/api/docs/guides/conversation-state

如果以后要做语音导游，官方文档把语音路线分成：

- 链式语音架构
- Realtime 实时语音架构

对这个项目，我的判断仍然是：

- 第一阶段先不要直接上实时语音
- 更适合先做文本导游 + TTS 讲解

来源：

- https://developers.openai.com/api/docs/guides/voice-agents
- https://developers.openai.com/api/docs/guides/realtime-webrtc
- https://developers.openai.com/api/docs/models/gpt-4o-mini-tts
- https://developers.openai.com/api/docs/models/gpt-realtime-1.5

### 体验设计方向判断

为了避免“AI 味”和通用聊天产品感，这次参考的方向是：

- UNESCO 的 `Dive into Heritage` 这种“地图/空间先行，自由探索”的方式  
  https://whc.unesco.org/en/dive-into-heritage/
- CyArk 这类数字遗产项目更强调“沉浸式、情境化讲述”，而不是只有资料列表  
  https://www.cyark.org/

基于这些方向，这个项目更适合做成：

- 开放探索
- 场景引导
- 路线陪伴
- 可随时介入但不强打断的导游

## 6. 当前最推荐的后端接法

### 第一阶段

后端只做文本导游：

- 接收前端上下文
- 使用后端自己的导游系统提示词
- 调用 `Responses API`
- 返回统一结构的 `reply`

### 第二阶段

在文本稳定后增加“播放讲解”：

- 后端先生成适合口播的短讲解
- 再调用 TTS 模型生成音频

### 第三阶段

最后再评估实时语音：

- 是否真的需要开麦
- 是否需要实时打断
- 是否已经把历史口径、路线逻辑、内容校验做稳

## 7. 已落地的接口准备

前端目前通过：

- `VITE_GUIDE_API_URL`
- `VITE_GUIDE_API_TOKEN`

连接导游后端。

请求中已新增：

- `activeRouteId`
- `guideBundle.routeCatalog`

这样后端能知道：

- 用户现在正在看哪里
- 用户大概在走哪条线
- 当前导游应当保持什么口吻

## 8. 最终建议

一句话总结：

**这个网站最适合做成“可自由漫游，但随时有一位懂景、懂路、懂节奏的导游轻轻接一句”的体验。**

所以接下来的优先顺序，仍然建议是：

1. 继续补每个建筑的导游短讲内容
2. 接真实文本导游 API
3. 增加可播放讲解
4. 最后再看是否做实时语音导游
